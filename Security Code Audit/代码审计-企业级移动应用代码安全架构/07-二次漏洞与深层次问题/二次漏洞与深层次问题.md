# 第 7 章 二次漏洞与深层次问题

> 本章对应主文档「第二部分 漏洞挖掘思路与防范」第 7 节，详细展开二次漏洞的成因、挖掘思路以及依赖与供应链安全。

---

## 7.1 什么是二次漏洞

### 7.1.1 定义

**二次漏洞**是指：攻击者先通过**第一个**漏洞或入口，将恶意数据**写入或篡改**到应用内部（如数据库、缓存、本地文件、配置）；然后该恶意数据在**另一个**功能或代码路径中被**读取并使用**，触发**第二个**漏洞（如 SQL 注入、XSS、越权、反序列化）。

- **特点**：写入点与触发点往往**不在同一处**，甚至不在同一模块；数据流经过「存储/持久化」再被读出，容易被误认为「来自内部、可信」而未经再次校验。
- **类比**：与 Web 中的「存储型 XSS」「二次 SQL 注入」类似：第一次把 payload 存进去，第二次在展示或查询时触发。

### 7.1.2 典型场景（移动端）

| 场景 | 第一次（写入/篡改） | 第二次（触发） |
|------|---------------------|----------------|
| 昵称/评论 | 用户输入恶意字符串存入 DB 或本地 | 另一处读取并拼接到 SQL、或输出到 WebView/页面 |
| 缓存/配置 | 通过某接口或越权写入恶意配置/缓存 | 启动或某功能读取配置并执行、拼接或反序列化 |
| 文件 | 通过上传或路径遍历写入恶意文件 | 其他模块读取该文件并解析、执行或包含 |
| 同步数据 | 服务端或恶意端下发恶意数据并写入本地 | 本地逻辑使用该数据做查询、跳转或执行 |

## 7.2 二次漏洞的审计思路

### 7.2.1 数据流追踪

- **从「写入」出发**：找出所有可被用户或外部间接控制的数据写入点（如昵称、评论、配置、缓存、文件、Intent 持久化）。
- **从「使用」出发**：找出所有从 DB、缓存、文件、配置中**读取**数据并参与敏感操作的地方（SQL、命令、反射、反序列化、WebView、Intent 启动）。
- **连起来**：判断「某处写入的数据」是否会在「另一处被读出并直接使用」而未做校验；若写入点可控、使用点危险，则存在二次漏洞可能。

### 7.2.2 信任边界

- **关键问题**：代码是否将「从存储/配置/网络同步来的数据」默认为**可信**，从而不再校验？
- **审计时**：凡是从「非当前请求直接传入」的来源（DB、文件、SharedPreferences、网络响应缓存）取出的数据，若参与敏感操作，都应视为「可能被篡改」，需做类型与白名单校验。

### 7.2.3 业务复杂处优先

- 购物车、订单、评论、草稿、用户资料、引用/转发等**与数据库或文件多次交互**的功能，更容易出现「先写后读再误用」的二次漏洞。
- 建议：先列出「写库/写文件/写配置」的入口与「读库/读文件/读配置并参与敏感操作」的 sink，再做矩阵式追踪。

---

## 7.3 依赖与供应链安全

### 7.3.1 风险来源

- **第三方 SDK / 库**：存在已知 CVE 或恶意行为；过度权限；数据外传；与主应用共享进程导致可被利用。
- **原生依赖**：.so / .a / .framework 中的漏洞或后门；版本过旧未修复。
- **KMP / 跨平台依赖**：Ktor、kotlinx.serialization、SQLDelight 等版本与配置不当；传递依赖引入的漏洞。

### 7.3.2 审计与排查方法

- **清单**：用 Gradle（`dependencies` 任务）、CocoaPods（`Podfile.lock`）、SPM（`Package.resolved`）导出依赖树，得到 group:name:version（或等价信息）。
- **CVE 与公告**：将依赖与 NVD、GitHub Advisory、OSV、各厂商安全公告比对；关注已知漏洞的版本范围与修复版本。
- **行为与权限**：对敏感 SDK（如推送、统计、支付、社交）阅读其文档与隐私政策；检查其声明的权限与网络请求是否与业务一致；必要时做动态或静态行为分析。

### 7.3.3 防范要点

- **依赖管理**：定期升级依赖；使用 Dependabot、Renovate 等自动化依赖更新与 CVE 提醒。
- **最小依赖**：仅引入必要依赖；优先选用维护活跃、安全响应及时的库。
- **构建与发布门禁**：在 CI 中集成依赖扫描（如 OWASP Dependency-Check、Snyk）；存在已知高危 CVE 的依赖禁止合入或发布，或要求有缓解方案与时间表。

---

## 7.4 小结

- **二次漏洞**：先通过一处漏洞或入口写入/篡改数据，再在另一处读取并误用该数据触发漏洞；审计时需追踪「数据从哪来、到哪去」，并审视「从存储/配置/网络来的数据」是否被当作可信使用。
- **供应链**：建立依赖清单、定期查 CVE 与安全公告、对敏感 SDK 做行为与权限审查；通过门禁与升级策略控制依赖风险。
