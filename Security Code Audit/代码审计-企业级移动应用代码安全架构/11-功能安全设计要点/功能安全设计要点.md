# 第 11 章 功能安全设计要点（移动端）

> 本章对应主文档「第三部分 安全编程规范与体系建设」第 11 节，按功能模块列出常见安全问题与安全设计要点。

---

## 11.1 功能与安全要点总表

| 功能 | 常见问题 | 安全设计要点 |
|------|----------|--------------|
| 登录/会话 | 弱密码、会话固定、Token 明文存储、撞库 | 强密码策略、服务端会话、Token 安全存储与刷新、限次与风控 |
| 找回密码/验证码 | 验证码可爆破、未绑定用户、凭证可预测、短信轰炸 | 限次、限时、绑定身份、随机强凭证、不泄露于前端 |
| 支付 | 金额/数量客户端可改、重复提交 | 金额与订单以服务端为准、幂等与锁、二次确认 |
| 文件/数据下载 | 路径遍历、任意文件读、未校验完整性 | 白名单路径、校验签名/哈希、不信任客户端路径 |
| 深链接/推送 | 未校验来源与参数、跳转劫持 | Scheme/链接白名单、参数校验、来源校验 |
| 生物识别 | 仅本地验证即通过、无服务端参与 | 与服务端二次校验、防重放、不替代密码存储 |
| **KMP 共享层** | 共享逻辑漏洞双端受影响、expect/actual 单端弱实现、平台入参未统一校验 | 优先审计 commonMain；expect/actual 双端等价安全；平台传入共享层的数据在共享层或两端统一校验 |

---

## 11.2 登录与会话

### 11.2.1 常见问题

- **弱密码**：无复杂度要求、允许常见弱口令，易被撞库。
- **会话固定**：登录后未重新生成 Session/Token，旧会话仍有效。
- **Token 明文存储**：Token 写在 SharedPreferences/UserDefaults 或普通文件，备份或 root 后可被窃取。
- **撞库**：无验证码或限次，可批量尝试账号密码。

### 11.2.2 安全设计要点

- **密码策略**：服务端强制复杂度（长度、大小写、数字、特殊字符）；客户端可做基础校验以提升体验。
- **会话**：登录成功后由服务端颁发新 Token/Session，旧 Token 失效；登出与过期时客户端清除本地 Token。
- **Token 存储**：使用 EncryptedSharedPreferences / Keychain；不写入日志与备份。
- **限次与风控**：登录失败限次、验证码、设备/IP 风控由服务端实施；客户端不单独依赖「本地限次」防撞库。

---

## 11.3 找回密码与验证码

### 11.3.1 常见问题

- **验证码可爆破**：位数过短、无错误次数与时间限制。
- **未绑定用户**：验证码或重置链接未与「待重置账号」强绑定，可被用于重置他人密码。
- **凭证可预测**：重置 Token 基于时间或简单规则生成，可预测。
- **短信/邮件轰炸**：发送接口无限流，可被滥用对某号码/邮箱轰炸。

### 11.3.2 安全设计要点

- **限次与限时**：单用户/单手机号/单 IP 限次；验证码与重置链接设短有效期（如 5–15 分钟）。
- **绑定身份**：验证码与「待重置账号」在服务端绑定；重置时校验「当前请求的 Token 是否属于该账号」。
- **随机强凭证**：Token 使用密码学安全随机数、足够长度（如 128 bit 以上）；不基于时间或用户 ID 简单生成。
- **不泄露于前端**：验证码与正确 Token 不放在 HTML/响应体明文、不通过客户端可读方式传递；发送后仅通过短信/邮件等独立通道告知用户。

---

## 11.4 支付

### 11.4.1 常见问题

- **金额/数量客户端可改**：订单金额、数量、优惠由客户端传入且服务端未严格校验，可篡改。
- **重复提交**：同一订单可多次支付或多次领取，无幂等与锁。

### 11.4.2 安全设计要点

- **服务端为准**：金额、数量、优惠、库存等由服务端根据订单与业务规则计算并持久化；客户端仅展示与提交「订单标识」或「确认意图」，不提交最终金额。
- **幂等与锁**：支付与扣款接口做幂等（如订单 ID + 幂等键）；对账户余额或库存加锁，避免并发重复扣款或超卖。
- **二次确认**：关键支付步骤可要求验证码、生物识别或服务端二次确认，并由服务端校验。

---

## 11.5 文件与数据下载

### 11.5.1 常见问题

- **路径遍历**：下载或读取文件时使用客户端传入的路径，未做规范化与白名单，可访问任意路径。
- **任意文件读**：通过参数指定「文件名」或「路径」直接读文件并返回，未限制在安全目录。
- **未校验完整性**：下载的安装包或资源未校验签名/哈希，可能被篡改。

### 11.5.2 安全设计要点

- **白名单路径**：仅允许访问应用私有目录或明确白名单目录；路径由服务端下发或由客户端根据「文件 ID」映射到安全路径，不直接使用用户可控路径。
- **校验完整性**：重要文件（如 APK/IPA、配置）下载后校验签名或哈希，与服务端公布值比对。
- **不信任客户端路径**：服务端若提供「按 ID 下载」接口，不根据客户端传入的路径读文件；由服务端根据 ID 查路径并返回内容。

---

## 11.6 深链接与推送

### 11.6.1 常见问题

- **未校验来源与参数**：任意 Scheme 或 URL 可调起 App 并传入参数；参数未校验即用于跳转、加载 URL 或执行业务逻辑。
- **跳转劫持**：从不可信页面或 App 通过深链接跳转到敏感页面并携带参数，诱导用户执行操作。

### 11.6.2 安全设计要点

- **Scheme/链接白名单**：仅处理业务声明的 Scheme 与域名（如 Universal Link 的 AASA）；不处理任意 URL。
- **参数校验**：对 path、query 做类型与业务白名单校验后再用于跳转或加载；不将参数直接拼接到 WebView URL 或内部路由。
- **来源校验**：若支持「从浏览器/其他 App 调起」，通过签名、Referrer 或服务端下发的一次性 Token 等方式校验来源（在可行范围内）。

---

## 11.7 生物识别

### 11.7.1 常见问题

- **仅本地验证**：生物识别通过后仅在本地「放行」，不通知服务端；攻击者可在本地绕过或重放。
- **无服务端参与**：敏感操作（如支付、改密码）仅依赖本地生物识别，服务端无二次校验。
- **替代密码存储**：用生物识别结果直接作为「密钥」或「密码」使用，存在重放与设备丢失风险。

### 11.7.2 安全设计要点

- **与服务端二次校验**：生物识别通过后，由客户端向服务端证明「当前设备已通过生物识别」（如通过已建立的安全通道提交签名或 Token），服务端再放行敏感操作。
- **防重放**：服务端下发 nonce 或挑战，客户端将生物识别结果与 nonce 绑定后提交，避免重放。
- **不替代密码**：生物识别用于「解锁本地密钥」或「确认当前用户」，不直接作为可存储的密码或密钥材料长期使用。

---

## 11.8 KMP 共享层

### 11.8.1 常见问题

- **共享逻辑漏洞双端受影响**：commonMain 中加解密、网络、序列化、存储等问题会同时影响 Android 与 iOS。
- **expect/actual 单端弱实现**：某一端 actual 使用弱加密、明文存储或未做证书 Pinning，形成单端漏洞。
- **平台入参未统一校验**：Android 或 iOS 传入共享层的数据未在共享层或两端统一校验，一端校验一端未校验导致绕过。

### 11.8.2 安全设计要点

- **优先审计 commonMain**：对共享层做与平台层同等级别的审计与依赖管理。
- **expect/actual 双端等价安全**：加解密与存储的 actual 均使用 Keystore/Keychain、EncryptedSharedPreferences 等；不在任一端弱化。
- **统一校验**：平台传入共享层的 URL、路径、用户输入等在**共享层**做统一校验，或在两端各自做**同一套**校验规则，避免遗漏一端。

---

## 11.9 小结

- 各功能模块应在设计阶段明确「谁说了算」（服务端 vs 客户端）、敏感数据如何存储与传输、入参如何校验。
- 登录、找回密码、支付、文件、深链接、生物识别、KMP 共享层均有对应常见问题与设计要点，可作评审与审计检查项。
