# 第 5 章 常见漏洞与防范（基础篇）

> 本章对应主文档「第二部分 漏洞挖掘思路与防范」第 5 节，详细展开不安全存储、组件暴露与 Intent 注入、弱加密与硬编码、不安全网络通信的挖掘与防范。

---

## 5.1 不安全的数据存储

### 5.1.1 漏洞现象

- 敏感数据（密码、Token、PII、银行卡号等）以**明文**或**弱加密**形式存储在：
  - Android：`SharedPreferences`、普通文件、外部存储、数据库未加密表。
  - iOS：`UserDefaults`、`plist` 文件、世界可读目录下的文件。
- 备份或 root/越狱后，攻击者可直接读取这些文件或键值，导致账号接管、身份盗用或合规风险。

### 5.1.2 挖掘方法

- **搜索**：`SharedPreferences`, `UserDefaults`, `writeToFile`, `FileOutputStream`, `FileWriter`, `NSUserDefaults.standard`, `FileManager.default.createFile`, 以及业务中与 `password`, `token`, `secret`, `cookie` 等相关的写入逻辑。
- **关注**：写入内容是否明文、是否使用 EncryptedSharedPreferences / Keychain；文件路径是否在应用私有目录、权限是否仅为当前应用可读。

### 5.1.3 防范要点

- **Android**：敏感数据使用 `EncryptedSharedPreferences` 或先加密再写入；密钥来自 Android Keystore；避免在外部存储或备份中包含敏感数据，或通过 `android:fullBackupContent` 排除。
- **iOS**：敏感数据存入 **Keychain**（kSecAttrAccessible 选用合适级别）；不将敏感信息写入 UserDefaults 或普通 plist。
- **KMP**：若在 commonMain 中封装存储，expect/actual 两端都应使用平台推荐安全 API（Keystore/Keychain、EncryptedSharedPreferences），不在共享层写死密钥。

## 5.2 组件暴露与 Intent 注入（Android）

### 5.2.1 漏洞现象

- **组件暴露**：Activity、Service、BroadcastReceiver 设置了 `android:exported="true"` 或未显式设为 false 且存在 `intent-filter`，可被其他应用调起。
- **Intent 注入**：调用方通过 Intent 的 extras 传入恶意数据（如路径、序列化对象、Deep Link 参数），目标组件未校验即用于文件操作、WebView、反射、启动其他组件等，导致越权、拒绝服务或逻辑漏洞。

### 5.2.2 挖掘方法

- **Manifest**：搜索 `exported`、`intent-filter`，列出所有可被外部调起的组件。
- **代码**：对每个导出组件，搜索 `getIntent()`, `getExtras()`, `getStringExtra`, `getParcelableExtra` 等，追踪 extras 中参数的使用位置（文件路径、URL、类名、序列化数据等）。
- **判断**：若参数来自 Intent 且未做白名单/类型/长度校验即参与敏感操作，则存在风险。

### 5.2.3 防范要点

- **最小化导出**：仅业务确需被外部调用的组件才设为 `exported="true"`，其余显式设为 `false`（Android 12+ 对带 intent-filter 的组件默认行为有变化，需显式声明）。
- **调用方校验**：对「仅允许本应用或可信应用调起」的组件，做 **签名/包名校验**（如 `getCallingPackage`、签名比对）。
- **Intent 参数**：对 extras 做**白名单与类型校验**；不将不可信数据直接用于文件路径、URL 加载、反射、反序列化。

## 5.3 弱加密与硬编码密钥

### 5.3.1 漏洞现象

- 使用 **DES、RC4、MD5/SHA1** 做敏感哈希或加密（如密码存储、签名校验），易被碰撞或暴力破解。
- **密钥、API Key、密码** 写死在代码、资源文件或构建配置中，反编译即可获取，导致加密形同虚设或接口被滥用。

### 5.3.2 挖掘方法

- **搜索**：`Cipher.getInstance`, `MessageDigest`, `MD5`, `SHA1`, `DES`, `RC4`, `AES`（关注是否 ECB、是否缺 IV）；字符串常量、配置文件中的 `key`, `secret`, `password`, `api_key`, `apikey`。
- **资源**：`strings.xml`、`res/values`、`.env`、`BuildConfig`、Info.plist 等中的敏感字符串。

### 5.3.3 防范要点

- **算法**：对称加密使用 **AES-GCM** 等带认证的模式；哈希使用 **SHA-256** 及以上；密码存储使用 **Argon2/bcrypt** 等抗暴力破解算法。
- **密钥**：密钥从 **Keystore（Android）/ Keychain（iOS）** 生成或注入，或由服务端在安全通道下下发；不在代码与资源中硬编码。
- **API Key**：若必须放在客户端，则仅用于可被滥用的非核心接口，并配合服务端限速、签名校验；核心密钥应放在服务端。

---

## 5.4 不安全的网络通信

### 5.4.1 漏洞现象

- 使用 **HTTP** 明文传输，或 **HTTPS 但关闭/放宽证书校验**（如信任所有证书、信任用户 CA），导致中间人可窃听或篡改请求/响应。
- 敏感接口（登录、支付、个人数据）在未加密或弱校验的通道上传输。

### 5.4.2 挖掘方法

- **搜索**：`http://`, `setHostnameVerifier`, `TrustManager`, `X509TrustManager`, `allowsArbitraryLoads`, `NSExceptionAllowsInsecureHTTPLoads`, `NSExceptionAllowsInsecureConnection`, `network_security_config`（Android 的 trust-user-certs、cleartext）。
- **配置**：Android 的 `network_security_config` 与 `AndroidManifest` 中的 `usesCleartextTraffic`；iOS 的 ATS 与 Info.plist 中的例外。

### 5.4.3 防范要点

- **全站 HTTPS**：生产环境不使用 HTTP；若确有本地或内网明文需求，需限定范围并文档化。
- **严格证书校验**：不信任用户安装的 CA（除非业务明确要求）；不重写 `TrustManager`/`HostnameVerifier` 使所有证书通过。
- **证书 Pinning**：对关键接口使用证书或公钥 Pinning，降低中间人风险（需做好证书轮换与降级策略）。

---

## 5.5 小结

- **存储**：敏感数据进加密存储或 Keychain/Keystore，避免明文与可备份的敏感文件。
- **组件与 Intent**：最小化导出、校验调用方、对 Intent 参数做白名单与类型校验。
- **加密与密钥**：强算法、密钥从安全存储或服务端来、禁止硬编码。
- **网络**：HTTPS + 严格证书校验 + 关键场景 Pinning。
