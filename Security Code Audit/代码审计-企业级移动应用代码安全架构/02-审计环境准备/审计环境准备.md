# 第 2 章 审计环境准备

> 本章对应主文档「第一部分 代码审计前的准备」第 2 节，详细说明 Android、iOS、KMP 及跨平台项目的环境与配置要点。

---

## 2.1 基础环境

### 2.1.1 Android

| 项目       | 说明 |
|------------|------|
| **IDE**    | Android Studio（推荐与项目一致的大版本），便于阅读 Gradle、Manifest 与源码。 |
| **SDK**    | 安装项目要求的 compileSdk / targetSdk 及最低 minSdk；多版本有助于发现「仅在某版本存在」的 API 或行为差异。 |
| **模拟器/真机** | 至少一种可运行环境，用于验证漏洞利用、网络与权限行为；真机可测试备份、证书安装等。 |
| **构建**   | 能成功执行 `./gradlew assembleDebug`（或项目既有脚本），确保依赖完整、无编译错误，便于搜索与跳转。 |

**审计注意**：不同系统/API 版本会影响 WebView、Intent、权限模型、存储路径等，漏洞利用条件可能因版本而异，建议在项目支持的最低与当前目标版本上各做一次关键验证。

### 2.1.2 iOS

| 项目       | 说明 |
|------------|------|
| **IDE**    | Xcode，用于打开 `.xcodeproj` / `.xcworkspace`、阅读 Info.plist、Entitlements、Swift/ObjC 源码。 |
| **模拟器/真机** | 至少一个可运行目标；真机需 Apple 开发者账号与签名，用于验证 Keychain、推送、关联域等。 |
| **构建**   | 能成功 Build 并运行，便于确认依赖（CocoaPods/SPM）与运行时代码路径。 |

**审计注意**：ATS、URL Scheme、Keychain 访问组、App Groups 等依赖配置与签名；模拟器与真机在部分安全 API 上行为可能不同，关键结论建议在真机复核。

### 2.1.3 KMP（Kotlin Multiplatform）

| 项目       | 说明 |
|------------|------|
| **工程结构** | 典型为多模块：`commonMain`（共享）、`androidMain` / `iosMain`（平台实现）；可能有 `commonTest` 等。 |
| **构建**   | Gradle 能成功编译 Android 与 iOS 目标（iOS 常通过 Kotlin/Native 或 KMP 的 CocoaPods/SPM 集成）。 |
| **双端验证** | 共享层逻辑需在 **Android 与 iOS 两端** 分别验证：同一段 Kotlin 在不同平台上的实际调用链与系统 API 可能不同。 |

**审计注意**：共享层漏洞会同时影响双端，应优先保证 `commonMain` 的构建与索引正常，并能在双端各做一次运行验证。

### 2.1.4 其他跨平台（RN、Flutter 等）

- **React Native**：Node 与 npm/yarn、对应 RN 版本、原生工程（Android Studio / Xcode）均可打开；审计时需同时看 JS/TS 与原生桥接、Native Module。
- **Flutter**：Flutter SDK、Dart 版本、能构建 Android/iOS；审计时关注 Dart 层与 Platform Channel、插件原生实现。

## 2.2 核心「配置」与约束

移动端没有类似 PHP 的单一 `php.ini`，但以下配置与约束会直接影响安全行为，审计时需逐一关注。

### 2.2.1 Android

#### AndroidManifest.xml

| 配置/属性           | 安全含义 |
|---------------------|----------|
| `android:exported`  | 组件是否可被外部应用调用；`true` 或无 `intent-filter` 时的默认值需重点看，易导致组件暴露与 Intent 注入。 |
| `android:permission` | 组件或权限的声明与使用是否匹配；是否声明了不必要的危险权限。 |
| `android:allowBackup` | 若为 `true`，需确认备份内容中是否包含敏感数据、是否通过 `android:fullBackupContent` 排除。 |
| `android:debuggable` | 发布包必须为 `false`；若为 `true` 则调试与注入风险高。 |
| `network_security_config` | 是否引用、是否放宽证书校验（如信任用户 CA、明文流量），与中间人风险直接相关。 |
| `usesCleartextTraffic` | 是否允许 HTTP 明文；应仅在确有需求且范围可控时使用。 |

#### build.gradle / build.gradle.kts

- **minSdk / targetSdk**：影响可用 API 与权限/行为（如 Scoped Storage、后台限制）。
- **混淆与 R8**：`minifyEnabled`、规则是否保留敏感类/方法（如反射、序列化用到的类），避免「混淆导致运行时异常」与「敏感逻辑被保留」的平衡。
- **依赖版本**：第三方库的 group:name:version，用于 CVE 与已知漏洞排查。

### 2.2.2 iOS

#### Info.plist

| 配置/Key            | 安全含义 |
|---------------------|----------|
| 权限描述（如相机、定位） | 是否与实际使用一致、是否存在过度申请。 |
| `URL Types` / URL Scheme | 是否可被其他 App 调起；Scheme 是否可猜测、参数是否校验。 |
| `NSAppTransportSecurity` | ATS 是否关闭、是否允许任意加载或例外域名，与中间人、明文流量相关。 |
| `ITSAppUsesNonExemptEncryption` | 导出合规相关；若为 false 需有合理依据。 |

#### Capabilities / Entitlements

- **Keychain Sharing**：Keychain 访问组是否过大、是否与其它 App 共享敏感数据。
- **App Groups**：共享容器与扩展间数据共享是否最小化、是否混入敏感数据。
- **Associated Domains**：Universal Links / AASA 配置是否仅限自家域名、防滥用。

### 2.2.3 通用

- **调试与日志**：发布构建是否关闭详细日志、是否包含 `Log.d`/`print` 输出敏感信息。
- **测试/调试代码**：是否包含 Mock、测试账号、调试开关、隐藏入口，发布前是否移除或关闭。
- **第三方 SDK**：版本、权限与行为（如数据采集、网络请求）、是否有已知 CVE 或下架风险。

### 2.2.4 KMP 特有配置与约束

| 项目         | 说明 |
|--------------|------|
| **模块结构** | `commonMain` 为共享代码；漏洞在此层则 **Android 与 iOS 同时受影响**，审计时应对共享层优先、全面。 |
| **expect/actual** | 平台相关 API（加密、存储、网络、证书）的声明与实现；需逐项核对各端 `actual` 是否使用安全 API、是否有硬编码或弱实现。 |
| **依赖**     | `build.gradle.kts` 中 `commonMain` 的 `sourceSets` 与依赖（如 Ktor、kotlinx.serialization、SQLDelight）；版本与安全配置（如禁用不安全序列化、启用证书 Pinning）。 |
| **导出与入口** | 共享模块对外暴露的接口；调用方为 Android Activity 或 iOS ViewController 时，是否在「共享层或两端」对入参做统一校验，避免越权或注入。 |

---

## 2.3 环境与配置检查清单（审计前）

- [ ] Android：Android Studio 可打开工程，Gradle 构建成功；Manifest 中 `exported`、`allowBackup`、`debuggable`、`network_security_config` 已查看。
- [ ] iOS：Xcode 可打开工程，Build 成功；Info.plist 中 ATS、URL Scheme、权限描述已查看；Entitlements 已查看。
- [ ] KMP：`commonMain` / `androidMain` / `iosMain` 结构清晰；expect/actual 列表与依赖已列出；能在双端各跑一遍关键流程。
- [ ] 通用：依赖列表已导出（如 `./gradlew dependencies`、CocoaPods）；调试/测试代码与敏感日志已初步排查。
