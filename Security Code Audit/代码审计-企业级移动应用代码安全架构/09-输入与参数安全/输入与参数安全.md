# 第 9 章 输入与参数安全

> 本章对应主文档「第三部分 安全编程规范与体系建设」第 9 节，详细说明移动端「输入不可信」原则与落地实践。

---

## 9.1 原则：所有外部输入均不可信

### 9.1.1 外部输入的定义

凡**非应用自身在受控环境中生成**的数据，均视为外部输入，包括但不限于：

| 来源 | 示例 |
|------|------|
| 用户输入 | 文本框、选择框、手势、生物识别结果（由系统返回，仍需校验） |
| Intent / 深链接 | Activity 的 getIntent()/extras、iOS openURL 参数、App Link / Universal Link 参数 |
| 推送 | 通知 payload、自定义 data 字段 |
| 剪贴板 | UIPasteboard、ClipboardManager 内容 |
| 文件 | 用户选择或外部传入的文件路径与内容、下载文件 |
| 网络响应 | API 返回的 JSON/XML、图片、HTML 等 |
| 本地存储的「历史数据」 | 若该数据曾由外部写入或可被其他漏洞篡改，再次读取时应视为不可信 |
| 第三方 SDK 回调 | 部分 SDK 会回调「来自外部」的数据（如分享结果、广告点击 URL） |

### 9.1.2 为什么不可信

- 调用方可伪造（如其他 App 调起你的组件并传入恶意 Intent）。
- 传输与存储可能被篡改（中间人、root/越狱后改文件或 DB）。
- 服务端或 SDK 可能被攻破或返回异常数据。
- 用户可能输入畸形、超长或恶意构造的字符串。

因此，**参与敏感操作前必须校验**，不能依赖「默认来自可信源」。

---

## 9.2 校验与白名单实践

### 9.2.1 类型与长度

- **类型**：数字、枚举、布尔等应做类型校验或安全转换（如 Android 的 `getIntExtra` 带默认值、iOS 的类型转换与可选解包）；字符串需明确「允许的字符集」与长度上限。
- **长度**：对字符串、数组长度做上限校验，避免溢出、DoS 或解析异常；长度限制应与业务与服务端一致。

### 9.2.2 业务规则

- **范围**：数字是否在合理区间（如金额 > 0、数量 ≤ 库存、状态枚举在已知集合内）。
- **格式**：手机号、邮箱、URL、ID 等是否符合约定格式；可用正则或格式库，但要注意 ReDoS 与边界情况。
- **关系**：参数之间的业务关系（如 orderId 与 userId 是否匹配）应在服务端校验；客户端可做基础格式与范围校验以提升体验。

### 9.2.3 白名单优于黑名单

- **白名单**：只允许「已知合法」的取值或模式，其余一律拒绝；例如 URL Scheme 白名单、文件扩展名白名单、操作类型白名单。
- **黑名单**：容易遗漏或绕过（编码、大小写、等价写法）；仅在无法枚举合法集时作为补充，且需与白名单结合（如先白名单再黑名单过滤危险字符）。

### 9.2.4 禁止直接参与的危险操作

以下操作**不得**在未校验或未白名单的情况下，直接使用外部输入作为输入：

- 拼接到 **SQL** 语句（应使用参数化/ORM）。
- 作为**命令**或**命令行参数**（应禁止或严格白名单）。
- 作为**反射**的类名/方法名（应禁止或严格白名单）。
- 作为**反序列化**的源（应禁止不可信反序列化或严格 schema 校验）。
- 作为**文件路径**（应规范化、白名单目录、禁止 `..` 与绝对路径等）。
- 作为 **WebView** 的 URL 或 `evaluateJavaScript` 的脚本内容（应 URL 白名单、禁止脚本注入）。
- 作为**启动组件**的类名或 Intent 的 Component（应白名单或由服务端下发受控列表）。

## 9.3 典型场景

### 9.3.1 Intent / 深链接参数

- **校验时机**：在入口处（Activity/ViewController、处理 openURL 的 delegate）统一校验，再传递给业务层；避免在业务层多处分散校验导致遗漏。
- **校验内容**：Scheme、path、query 参数的类型与范围；若参数用于跳转、加载 URL、打开文件，必须做白名单或严格业务校验。
- **KMP**：若参数由平台层传入共享层，应在**共享层或两端统一**做同一套校验，避免一端校验一端未校验。

### 9.3.2 WebView 与 Scheme

- **URL**：仅加载白名单域名或 path；不将用户输入或 Intent 参数直接拼接到 base URL 后加载，除非经过编码与白名单校验。
- **Scheme**：仅处理业务需要的 Scheme；参数解析后做类型与业务校验再使用。
- **evaluateJavaScript**：不将用户输入或网络数据直接传入；若必须动态生成脚本，使用严格的模板或 JSON 传参并由前端按需渲染。

### 9.3.3 网络响应

- **解析**：使用安全解析方式（如标准 JSON/XML 解析器），避免将响应体当作代码执行（如不安全的反序列化、`eval` 等）。
- **使用前**：对影响安全逻辑的字段（如 URL、路径、状态、权限相关）做类型与业务校验；不默认信任「服务端一定正确」。

---

## 9.4 小结

- **所有外部输入均不可信**：用户输入、Intent、推送、剪贴板、文件、网络、存储历史数据、第三方回调等，参与敏感操作前必须校验。
- **实践**：类型与长度、业务规则、白名单优先；禁止不可信输入直接参与 SQL、命令、反射、反序列化、文件路径、WebView URL/脚本等。
- **入口统一**：在入口或共享层做统一校验，避免遗漏；KMP 下平台传入共享层的数据应在共享层或两端统一校验。
