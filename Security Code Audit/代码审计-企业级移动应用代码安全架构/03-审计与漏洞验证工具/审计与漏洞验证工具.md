# 第 3 章 审计与漏洞验证工具

> 本章对应主文档「第一部分 代码审计前的准备」第 3 节，展开静态分析、抓包与调试、编码与检索等工具的使用场景与注意点。

---

## 3.1 静态分析工具

### 3.1.1 工具一览与选型

| 工具 | 平台/语言 | 主要用途 | 说明 |
|------|-----------|----------|------|
| **MobSF** | Android / iOS | 一键静态分析 | 上传 APK/IPA 或源码，自动检测敏感 API、权限、硬编码、依赖漏洞、证书与网络配置等；适合快速建立风险视图，结果需人工复核。 |
| **Qark** | Android | 规则化漏洞扫描 | 针对 Android 的 Manifest、组件、Intent、存储、加密等做规则检测；可集成到 CI。 |
| **SpotBugs** | Java/Bytecode | 缺陷与坏味道 | 含安全相关规则（如硬编码密码、弱随机数）；可与 Android Gradle 集成。 |
| **Semgrep** | 多语言（含 Kotlin） | 自定义规则、漏洞模式 | 编写规则匹配「危险 API + 可控输入」等模式；适合团队沉淀审计规则与 KMP 共享层。 |
| **SonarQube** | 多语言 | 代码质量与安全 | 安全规则、坏味道、重复代码；需配置规则集与质量门禁。 |
| **Detekt / ktlint** | Kotlin | 代码风格与部分安全 | Kotlin 专用；可配合自定义规则做 KMP 共享层基础扫描。 |
| **IDB / Frida** | iOS | 逆向与动态插桩 | 越狱/非越狱环境下调试、Hook、脱壳；在授权范围内用于漏洞验证与深度分析。 |

### 3.1.2 使用注意

- **误报**：静态工具常有误报（如误判「可控」、误判「敏感」），结论需结合数据流与业务逻辑人工确认。
- **漏报**：业务逻辑漏洞、设计缺陷、多步组合利用，工具难以全覆盖，需结合审计思路（第 4 章）与人工通读。
- **KMP**：共享层一处漏洞双端受影响，对 `commonMain` 的审计优先级应提高；可用 Semgrep/Detekt 针对 Kotlin 写规则，双端仍用 MobSF、Xcode 等做平台侧补充。

## 3.2 抓包与调试

### 3.2.1 HTTPS 抓包

| 工具 | 说明 |
|------|------|
| **Charles / Fiddler / Burp Suite** | 代理方式抓取 App 的 HTTP/HTTPS；需在设备上安装代理证书并信任，必要时配合绕过证书 Pinning（仅限授权测试）。 |
| **配置要点** | 手机与电脑同网、代理指向电脑 IP 与端口；Android 7+ 及高版本 iOS 需应用信任用户 CA 或修改 App（如 debug 构建中禁用 Pinning）。 |

**审计用途**：确认请求/响应是否加密、参数是否可篡改、是否存在敏感信息泄露；验证逻辑漏洞（如改金额、改状态）是否在服务端校验。

### 3.2.2 系统与进程调试

| 工具/命令 | 说明 |
|-----------|------|
| **adb** | Android：日志（logcat）、文件拉取、进程列表、备份；便于查看运行时路径、权限与错误信息。 |
| **LLDB / Xcode Debugger** | iOS：断点、变量查看、控制台输出；真机需配置签名与开发者模式。 |
| **Objection / Frida** | 动态插桩：Hook 函数、绕过 Root/越狱检测与证书校验；需在授权范围内使用。 |

**审计用途**：验证漏洞是否可利用、关键参数在运行时的真实来源与取值；分析加固/混淆后的行为。

### 3.2.3 脱壳与逆向（授权范围内）

- **Android**：脱壳工具与 Frida 脚本用于还原 DEX；审计时若仅有 APK，需先还原再结合反编译结果阅读逻辑。
- **iOS**：脱壳、class-dump、IDA/Hopper 等；用于确认二进制中的类与方法、第三方库版本与符号。

## 3.3 编码与检索

### 3.3.1 IDE 与项目打开方式

- **Android**：Android Studio 打开项目根目录（含 `build.gradle`），保证 Gradle 同步成功，便于全局搜索与符号跳转。
- **iOS**：Xcode 打开 `.xcworkspace`（若用 CocoaPods）或 `.xcodeproj`；可同时用 VS Code 等打开 Swift/ObjC 做全文搜索。
- **KMP**：Android Studio 打开 KMP 工程根目录，确保 `commonMain`、`androidMain`、`iosMain` 均被索引。

### 3.3.2 全局搜索关键字

审计时常用「敏感 API + 可控输入」思路，搜索关键字参见主文档第 4.1 节与第 5、6 章，例如：

- **存储与密钥**：`SharedPreferences`、`UserDefaults`、`Keychain`、`EncryptedSharedPreferences`、`password`、`token`、`secret`、`key`。
- **网络**：`http://`、`TrustManager`、`HostnameVerifier`、`allowsArbitraryLoads`、`NSExceptionAllowsInsecureHTTPLoads`。
- **组件与 Intent**：`exported`、`getIntent()`、`getExtras()`、`startActivity`、`sendBroadcast`。
- **加密与哈希**：`Cipher.getInstance`、`MessageDigest`、`MD5`、`SHA1`、`DES`、`RC4`。
- **WebView 与深链接**：`WebView`、`loadUrl`、`evaluateJavascript`、`addJavascriptInterface`、`openURL`、URL Scheme。
- **KMP**：`expect`/`actual`、`HttpClient`、`decodeFromString`、`SqlDriver`、`KeyValueStore`。

可结合正则（如 `key\s*=\s*["'][^"']+["']` 找硬编码字符串）与「在路径中排除 test、mock、sample」提高效率。

### 3.3.3 依赖与 CVE 排查

- **Android**：`./gradlew :app:dependencies`（或对应模块）导出依赖树；将 group:name:version 与 NVD、GitHub Advisory 等比对。
- **iOS**：CocoaPods `Podfile.lock`、SPM `Package.resolved` 中的依赖与版本；逐一查 CVE 与安全公告。
- **KMP**：`commonMain` 与各端依赖（如 Ktor、kotlinx.serialization、SQLDelight）的版本与安全配置。

---

## 3.4 工具使用流程建议

1. **静态扫描**：用 MobSF（或等价工具）对 APK/IPA 或源码做一轮扫描，形成「敏感 API、权限、依赖、配置」清单。
2. **关键字与数据流**：按审计思路（第 4 章）对清单与关键功能做关键字搜索与数据流追踪，标注疑似点。
3. **抓包与运行**：对疑似点做抓包、改参、重放，确认是否可利用；必要时用 Frida/LLDB 做运行时验证。
4. **依赖与 CVE**：对依赖树做 CVE 与已知漏洞排查，与代码审计结论合并成报告。
