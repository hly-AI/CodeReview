# 第 8 章 审计中的小技巧（移动端）

> 本章对应主文档「第二部分 漏洞挖掘思路与防范」第 8 节，整理审计过程中常用的思路与注意点：信任边界、编码与截断、并发与多进程等。

---

## 8.1 「可信数据」的等价概念（对应 Web 的 GPC）

### 8.1.1 背景

- Web 中曾有「魔术引号」（GPC）等对输入做自动转义；移动端没有统一等价物，但存在**「默认信任某类数据」**的假设，若被打破就会出问题。
- **审计关注**：哪些数据被开发者**默认为可信**，从而未做校验即参与敏感操作？

### 8.1.2 常见「被误认为可信」的数据

| 来源 | 风险 |
|------|------|
| **服务端下发的数据** | 若被篡改（中间人、恶意服务端、缓存污染），直接参与 SQL、命令、反射、反序列化、URL 加载等会触发漏洞。 |
| **本地存储的数据** | 若被其他漏洞或 root/越狱后篡改，再次读取时未校验即使用，可能导致二次漏洞。 |
| **Intent / 深链接参数** | 调用方可控，未校验即用于文件路径、WebView、启动组件等。 |
| **来自「本应用其他模块」的数据** | 若该模块存在越权或注入，传入的数据可能已被污染。 |

### 8.1.3 审计要点

- 对「敏感 sink」（SQL、命令、反射、反序列化、文件路径、WebView URL）的**每个输入**做溯源：若来自上述任一来源，应检查是否有**显式校验**（类型、长度、白名单、业务规则）；若无，则记入疑似点并验证。

## 8.2 编码与截断

### 8.2.1 文件名与路径

- **路径遍历**：用户或 Intent 传入的文件名/路径中包含 `../` 或编码形式（如 `%2e%2e%2f`），若未做规范化与白名单校验，可能访问到应用私有目录外文件。
- **截断**：历史上有通过 `%00`、长路径等截断文件名或路径的案例；审计时关注「文件路径是否来自外部」以及「是否做了长度与字符集限制」。

### 8.2.2 URL 与 Intent 参数

- **编码**：URL 参数、Intent extras 可能经过多次编码；解码顺序与次数若与校验逻辑不一致，可能绕过校验（如先校验再解码、或只校验一种编码）。
- **长度与格式**：超长字符串、畸形 Unicode、控制字符可能导致解析差异或截断，需结合业务看是否对长度与字符集做限制。

### 8.2.3 审计建议

- 对「接受外部路径或 URL」的 API，确认：是否做**规范化**（resolve、去掉 `..`）、是否**限制在应用私有或白名单目录**、是否对长度与非法字符做校验。
- 对 Intent、深链接、WebView URL 的参数，确认解码与校验的**顺序**与**一致性**（如统一在解码后校验一次）。

---

## 8.3 多进程与多线程

### 8.3.1 敏感状态一致性

- **多进程**：Android 下多进程组件（如 `android:process` 独立进程）共享数据需通过 Binder、文件或 DB；若某进程未正确校验或加锁，可能导致越权或状态不一致。
- **多线程**：同一进程内多线程访问共享变量（如「是否已支付」「剩余次数」）若未做同步，可能导致竞态（如重复扣款、重复领取）。

### 8.3.2 审计要点

- **支付、订单、扣款、领取**等「关键状态」的更新与读取是否在**服务端**完成；若在客户端，是否对并发请求做**幂等与锁**（如服务端分布式锁、乐观锁）。
- **本地标志位**（如「是否已弹窗」「是否已执行某操作」）若仅存于内存且无锁，在高并发或快速点击下可能被绕过；需结合业务判断是否需服务端参与或本地加锁。

---

## 8.4 其他小技巧

### 8.4.1 日志与调试信息

- **发布包**：确认是否关闭详细日志、是否包含 `Log.d`/`print` 输出敏感信息（Token、密码、个人数据）；搜索 `Log.`, `println`, `print(`, `NSLog` 等。
- **调试接口**：是否存在仅通过「特定 Build 或配置」开启的调试接口、测试账号、Mock 数据；发布前是否移除或关闭。

### 8.4.2 版本与分支

- **条件编译 / 配置**：不同 buildType、flavor 或环境可能启用不同逻辑（如 debug 关闭证书校验）；审计时需明确**发布版本**对应的配置与代码路径。
- **遗留代码**：历史遗留的「备用逻辑」「兼容旧版」可能包含弱校验或已废弃的敏感接口；建议对未使用的分支做标注或清理。

### 8.4.3 第三方与系统行为

- **系统/厂商定制**：不同厂商、ROM 对 Intent、权限、后台行为的实现可能不同；关键结论尽量在**多设备/多版本**上验证。
- **第三方 SDK 回调**：传入 SDK 的数据与从 SDK 回调获得的数据，是否被当作可信使用；SDK 文档是否声明了「不可信」的入参或回调。

---

## 8.5 小结

- **信任边界**：明确「默认可信」的数据来源，对参与敏感操作的输入做显式校验。
- **编码与截断**：路径与 URL 做规范化与白名单；注意解码顺序与一致性；长度与非法字符限制。
- **并发**：关键状态以服务端为准；本地状态注意多进程/多线程下的同步与幂等。
- **日志与配置**：发布包关闭敏感日志与调试接口；审计时明确发布版本对应的代码与配置。
