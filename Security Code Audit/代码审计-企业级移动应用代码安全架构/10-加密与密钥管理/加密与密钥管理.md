# 第 10 章 加密与密钥管理

> 本章对应主文档「第三部分 安全编程规范与体系建设」第 10 节，详细说明移动端加密算法选型、密钥存储与传输安全。

---

## 10.1 密钥与长期凭证的存储

### 10.1.1 原则

- **不写死**：密钥、API Key、密码等不得硬编码在代码、资源文件或构建配置中；反编译即可获取，加密形同虚设。
- **进安全存储**：长期凭证与密钥应存放在** Android Keystore** 或 **iOS Keychain**，利用系统提供的硬件/安全 enclave 保护（在支持的设备上）。
- **访问控制**：Keychain/Keystore 的访问控制（如仅当前应用、仅解锁后可用）需与业务一致；避免使用「始终可访问」级别存高敏感数据。

### 10.1.2 Android

- **Keystore**：使用 `AndroidKeystore` 生成或导入密钥；密钥材料不离开安全硬件（在支持 TEE/StrongBox 的设备上）。
- **EncryptedSharedPreferences**：若需存储键值型敏感数据（如 Token），使用 `EncryptedSharedPreferences`，其密钥由 Keystore 保护。
- **备份**：避免将 Keystore/Keychain 或明文敏感数据包含在 `allowBackup` 的备份中；或通过 `fullBackupContent` 排除。

### 10.1.3 iOS

- **Keychain**：使用 Keychain Services 存储密钥与敏感数据；`kSecAttrAccessible` 选用合适级别（如 `kSecAttrAccessibleWhenUnlockedThisDeviceOnly`）。
- **Keychain Sharing**：仅在与业务需要的 App/扩展之间共享；访问组不宜过大，避免敏感数据被非预期应用读取。
- **UserDefaults / plist**：不存储密钥、密码、长期 Token；仅存非敏感的配置与 UI 状态。

### 10.1.4 KMP

- **expect/actual**：加解密与存储的「实际实现」在各端 actual 中，应统一使用 Keystore/Keychain、EncryptedSharedPreferences 等平台推荐 API；不在 commonMain 中写死密钥或实现「自己写文件加密」的弱方案。
- **密钥生成**：若在共享层定义「密钥用途与算法」，实际生成与存储应由各端 actual 调用平台 API 完成。

---

## 10.2 算法选型

### 10.2.1 禁止或避免使用的算法

| 用途 | 不推荐 | 原因 |
|------|--------|------|
| 对称加密 | DES、3DES、RC4、ECB 模式 | 弱算法或易被模式分析；ECB 无 IV，相同明文产生相同密文。 |
| 哈希（安全相关） | MD5、SHA-1 | 碰撞与长度扩展已不安全；仅可用于非安全场景（如缓存 key、校验和）。 |
| 密码存储 | 明文、MD5/SHA 单次哈希、可逆加密 | 易破解或拖库后还原；密码存储应使用专门算法。 |

### 10.2.2 推荐用法

| 用途 | 推荐 |
|------|------|
| 对称加密 | **AES-GCM**（或 AES-CCM）等带认证的模式；正确使用 IV/Nonce，且不重复使用同一 Key+IV。 |
| 哈希（完整性/指纹） | **SHA-256** 及以上（SHA-384、SHA-512）；若需 HMAC 则使用 HMAC-SHA256 等。 |
| 密码存储 | **Argon2**、**bcrypt**、**PBKDF2**（高迭代次数）；由服务端完成存储与校验，客户端仅提交哈希或通过安全协议传输。 |
| 非对称（密钥交换/签名） | **RSA**（足够长度）、**ECDH**、**Ed25519** 等；遵循各平台与库的推荐用法。 |

### 10.2.3 随机数

- **密钥、IV、Nonce、Token** 等必须使用**密码学安全随机数**（如 `SecureRandom`、`SecRandomCopyBytes`），不得使用 `Math.random()`、`Random()` 等伪随机。
- **KMP**：若在 commonMain 中生成随机数，应通过 expect/actual 调用各端安全随机 API，不在共享层使用不安全的伪随机。

## 10.3 传输安全

### 10.3.1 HTTPS

- **全站 HTTPS**：生产环境所有接口使用 TLS；不在客户端关闭证书校验或信任用户 CA（除非有明确且受限的调试场景，且不随发布包发布）。
- **TLS 版本与套件**：使用 TLS 1.2 及以上；禁用弱套件；遵循各平台与库的推荐配置。

### 10.3.2 证书 Pinning

- **适用场景**：高敏感 App（如金融、支付、企业内网）可对关键接口做**证书 Pinning** 或**公钥 Pinning**，降低中间人风险。
- **注意**：需制定证书轮换与过期策略；可考虑「多证书/多公钥」或「备份 pin」以避免单点过期导致不可用；调试时可仅在 debug 构建中禁用 Pinning。

### 10.3.3 不自造「加密协议」

- 不在应用层用「自定义对称加密」替代 TLS（如「自己 AES 加密 HTTP body」而不用 HTTPS）；传输层安全应交给 TLS，应用层可在此基础上做端到端加密等增强。
- 若业务确需端到端加密，应在 TLS 之上、使用标准算法与密钥管理，并由安全评审通过。

---

## 10.4 小结

- **存储**：密钥与长期凭证进 Keystore/Keychain；不写死、不放在普通文件或 Preferences。
- **算法**：对称用 AES-GCM；哈希用 SHA-256 及以上；密码存储用 Argon2/bcrypt 等；随机数用安全随机数生成器。
- **传输**：全站 HTTPS、严格证书校验、关键场景 Pinning；不以自定义加密替代 TLS。
