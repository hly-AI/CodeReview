# 代码审计：企业级移动应用代码安全架构

> 面向移动开发 Leader，基于《代码审计：企业级 Web 代码安全架构》思路，适配 iOS / Android 原生、**KMP（Kotlin Multiplatform）** 及跨平台应用的代码审计与安全设计。

---

## 文档说明

本文档将 Web 代码审计的体系（审计前准备 → 漏洞挖掘思路与防范 → 安全编程规范）迁移到**移动应用**场景，涵盖：

- **审计环境与工具**：移动构建环境、静态分析工具、抓包与调试工具（含 **KMP** 项目结构）
- **审计思路**：敏感 API 回溯、数据流追踪、功能点定向审计（含 **KMP 共享层与 expect/actual**）
- **常见漏洞**：不安全存储、组件暴露、网络与加密、逻辑与业务漏洞（含 **KMP 特有风险**）
- **安全编程规范**：输入校验、安全通信、权限与隐私、应用安全体系建设

适用对象：移动开发 Leader、架构师、安全工程师、负责 App 安全与合规的团队成员。

## 第一部分 代码审计前的准备

### 1. 为什么需要移动端代码审计

- **企业安全与合规**：满足等保、行业监管、应用市场上架安全要求。
- **SDL 在移动端的落地**：需求 → 设计 → 编码 → 测试 → 发布 → 运维，安全贯穿全生命周期；白盒审计（代码审计）是测试阶段的核心手段之一。
- **渗透与应急**：拿到脱壳/反编译包或源码时，能快速定位漏洞、评估影响、给出修复方案。

移动端代码审计对**编程基础**有要求：至少要能读懂业务逻辑、数据流、与系统/网络的交互；语言上以 **Java/Kotlin（Android）、Swift/Objective-C（iOS）** 为主，跨平台则涉及 **KMP（Kotlin Multiplatform）、React Native、Flutter、Cordova** 等。

### 2. 审计环境准备

#### 2.1 基础环境

| 类型 | 说明 |
|------|------|
| **Android** | Android Studio、多版本 SDK/模拟器（建议覆盖最低支持版本与最新版本）、真机 |
| **iOS** | Xcode、多版本模拟器、真机（需 Apple 开发者账号） |
| **KMP** | Kotlin 多平台工程：`commonMain` / `androidMain` / `iosMain` 源码集，Gradle 与 CocoaPods/SPM 构建，需同时能编译 Android 与 iOS 目标 |
| **其他跨平台** | 对应框架的 CLI 与依赖（如 RN、Flutter 的稳定版本） |

不同系统/API 版本会影响漏洞利用（如 WebView 漏洞、Intent 处理、权限模型），建议在多个版本下验证。**KMP 项目**需在 Android 与 iOS 两端分别验证共享逻辑的实际行为。

#### 2.2 核心“配置”与约束（对应 PHP 的 php.ini）

移动端没有单一“配置文件”，但以下等价于“环境约束”，审计时需心中有数：

- **Android**
  - `AndroidManifest.xml`：权限、组件导出、备份、网络安全配置（`network_security_config`）、`allowBackup`、`debuggable` 等。
  - `build.gradle`：minSdk/targetSdk、混淆与 R8、依赖版本。
- **iOS**
  - `Info.plist`：权限描述、URL Scheme、ATS、App Transport Security 例外。
  - Capabilities、Entitlements：Keychain、App Groups、Associated Domains 等。
- **通用**
  - 是否开启调试/日志、是否包含测试/调试代码、第三方 SDK 版本与已知 CVE。
- **KMP（Kotlin Multiplatform）**
  - **模块结构**：`commonMain`（共享代码）、`androidMain` / `iosMain`（平台实现）；共享层漏洞会同时影响双端，需优先审计。
  - **expect/actual**：平台 API 的声明与实现（如加密、存储、网络）；审计时需对照各端 actual 实现，确认无弱实现或硬编码。
  - **依赖**：`build.gradle.kts` 中 `commonMain` 的 `sourceSets`、KMP 库版本（如 Ktor、kotlinx.serialization、SQLDelight）；第三方 KMP 库的 CVE 与安全配置。
  - **导出与入口**：共享模块暴露的接口是否被 Android Activity/ iOS ViewController 正确校验后调用；避免“共享层信任调用方”导致的越权或注入。

### 3. 审计与漏洞验证工具

#### 3.1 静态分析工具

| 工具 | 平台 | 用途 |
|------|------|------|
| **MobSF** | Android / iOS | 一键静态分析、敏感 API、权限、硬编码、依赖漏洞 |
| **Qark / SpotBugs** | Android | 规则化漏洞与坏味道 |
| **Semgrep** | 多语言 | 自定义规则、框架与漏洞模式 |
| **SonarQube** | 多语言 | 代码质量与安全规则 |
| **IDB / Frida** | iOS | 逆向与动态插桩 |
| **KMP 项目** | Kotlin | 重点审计 `commonMain`；可用 Detekt、ktlint、Semgrep 规则针对 Kotlin；双端各自仍用平台工具（MobSF、Xcode 等） |

不能只依赖工具：误报需人工确认，业务逻辑漏洞、设计缺陷往往需通读代码。**KMP** 下共享代码一处漏洞双端受影响，审计优先级应提高。

#### 3.2 抓包与调试

- **Charles / Fiddler / Burp Suite**：HTTPS 抓包（需装证书、配置代理）。
- **adb / LLDB**：日志、进程、调试。
- **Objection / Frida**：脱壳、Hook、绕过证书校验与 Root/越狱检测（在授权范围内使用）。

#### 3.3 编码与检索

- **IDE**：Android Studio、Xcode、VS Code（跨平台项目）。
- **全局搜索**：敏感关键字（见下文审计思路）、硬编码密钥、URL、正则与加解密调用。

## 第二部分 漏洞挖掘思路与防范

### 4. 通用代码审计思路（移动端）

与 Web 审计类似，可归纳为四种：

1. **敏感 API / 关键字回溯**  
   从“危险调用”反推输入来源，判断是否可控、是否校验（如：加解密、文件读写、Intent、WebView、网络请求、SQL、反射、JNI）。
2. **可控变量正向追踪**  
   从 Activity/Fragment 入参、Intent、用户输入、SharedPreferences/UserDefaults、剪贴板等出发，跟踪到系统 API 或业务敏感逻辑。
3. **敏感功能点定向审计**  
   登录、支付、找回密码、分享、文件上传/下载、深链接、推送、本地存储、生物识别等，按功能逐块读代码。
4. **通读关键路径**  
   从启动页/主入口 → 登录 → 核心业务流程，理解整体架构与数据流，再结合上述三种方式挖洞。

**KMP 架构下的审计侧重**：

- **先审共享层（commonMain）**：加解密、网络请求、序列化/反序列化、本地存储封装、Token 处理等若在共享代码中实现，一旦有问题会同时影响 Android 与 iOS，应优先做敏感 API 回溯与数据流追踪。
- **expect/actual 对照审**：对 `expect` 的每个平台 `actual` 实现做核对，确认密钥/证书/存储路径等无弱实现或平台差异导致的单端漏洞（如 Android 用 Keystore、iOS 用 Keychain，实现是否等价安全）。
- **入口在各自平台**：Intent、深链接、剪贴板、文件访问等仍由 Android/iOS 端接收入参，再传入 KMP 层；审计时追踪“平台入口 → 共享层”的校验是否在共享层或平台层统一完成，避免遗漏一端校验。

#### 4.1 敏感关键字示例（便于搜索）

- **Android**：`startActivity`/`sendBroadcast`、`getSharedPreferences`、`openFileOutput`、`Runtime.exec`、`loadLibrary`/JNI、`WebView.loadUrl`/`evaluateJavascript`、`SQLiteDatabase.rawQuery`/`execSQL`、`Cipher.getInstance`、`MessageDigest`、`Intent`、`Parcelable`、`decodeFile`/`decodeStream`。
- **iOS**：`UserDefaults`、`Keychain`、`NSFileManager`、`NSURLSession`/`URLSession`、`WKWebView.evaluateJavaScript`、`sqlite3_*`、`CCCrypt`、`SecItemCopyMatching`、`UIApplication.openURL`、`UIPasteboard`。
- **KMP（commonMain / 共享 Kotlin）**：`expect`/`actual`、`HttpClient`（Ktor）、`serialization`/`decodeFromString`、`SqlDriver`（SQLDelight）、`KeyValueStore`/`Settings`、`Encryptor`/`Hash`、`UUID`/`random`、平台注入的 `HttpEngine`/`证书 Pinning`；以及共享层里所有接收“来自平台”的字符串/字节处（防止注入与反序列化风险）。
- **通用**：`password`、`token`、`secret`、`key`、`api_key`、`auth`、`cookie`、`http://`（非 HTTPS）、`eval`、`execute`、`decode`、`deserialize`。

### 5. 常见漏洞与防范（基础篇）

#### 5.1 不安全的数据存储

- **现象**：敏感数据（密码、Token、PII）写在 SharedPreferences/UserDefaults 未加密、或放在世界可读文件中。
- **挖掘**：搜 `SharedPreferences`、`UserDefaults`、`writeToFile`、`FileOutputStream` 等，看写入内容与权限。
- **防范**：密钥/Token 进 Keychain（iOS）或 EncryptedSharedPreferences/Keystore（Android）；文件存敏感信息时加密并控制权限。

#### 5.2 组件暴露与 Intent 注入（Android）

- **现象**：Activity/Service/BroadcastReceiver 导出且未做调用方校验，通过 Intent 传入恶意数据导致越权、拒绝服务或逻辑漏洞。
- **挖掘**：查 `AndroidManifest.xml` 中 `exported="true"` 或无 `intent-filter` 默认导出的组件；读组件的 `getIntent()`/`getExtras()` 使用处。
- **防范**：最小化导出；对调用方做签名/包名校验；对 Intent 参数做白名单与类型校验。

#### 5.3 弱加密与硬编码密钥

- **现象**：使用 DES、RC4、MD5/SHA1 做敏感哈希、或密钥写死在代码/资源中。
- **挖掘**：搜 `Cipher.getInstance`、`MessageDigest`、`MD5`、`SHA1`、字符串常量中的 key/secret。
- **防范**：AES-GCM 等强算法、密钥从 Keystore/Keychain 或服务端下发、密码用 Argon2/bcrypt 等。

#### 5.4 不安全的网络通信

- **现象**：HTTP 明文、证书校验被关闭、中间人可篡改/窃听。
- **挖掘**：搜 `http://`、`setHostnameVerifier`、`TrustManager`、`allowsArbitraryLoads`、`NSExceptionAllowsInsecureHTTPLoads`。
- **防范**：全站 HTTPS、严格证书校验、Pinning（证书/公钥）、不信任用户 CA。

### 6. 常见漏洞与防范（进阶篇）

#### 6.1 WebView 与前端协议

- **现象**：`addJavascriptInterface` 暴露敏感接口、`file://` 访问、任意 URL 加载、未校验 Scheme/深链接参数。
- **挖掘**：搜 WebView 相关 API、`@JavascriptInterface`、`shouldOverrideUrlLoading`、URL Scheme 处理。
- **防范**：最小化 JS 接口、白名单 URL/Scheme、禁用不必要的 file 访问。

#### 6.2 本地 SQL 注入与命令执行

- **现象**：SQL 语句拼接用户输入（包括从 Intent/文件来的）、`Runtime.exec`/ProcessBuilder 参数可控。
- **挖掘**：搜 `rawQuery`、`execSQL`、拼接 SQL 的字符串、`Runtime.getRuntime().exec`、`ProcessBuilder`。
- **防范**：参数化查询/ORM；命令/参数白名单，禁止用户输入直接进命令行。

#### 6.3 逻辑与业务漏洞

- **现象**：支付金额/数量从客户端传入未校验、找回密码/验证码可爆破或未绑定用户、投票/抽奖次数依赖客户端或可伪造。
- **挖掘**：通读支付、找回密码、验证码、投票/活动相关代码，看“谁说了算”（服务端 vs 客户端）。
- **防范**：关键结果以服务端为准；敏感操作多因素校验；限速、限次数、绑定身份。

#### 6.4 KMP 架构相关风险

- **现象**：共享层（commonMain）中的加解密、网络、序列化、存储等存在弱实现或硬编码；某端 `actual` 实现与另一端安全等级不一致；平台传入共享层的数据未在共享层或平台层统一校验，导致双端或单端可利用。
- **挖掘**：重点审 `commonMain` 下所有接触“外部输入”或“平台传入”的模块（Ktor HttpClient、kotlinx.serialization、SQLDelight、自定义 KeyValueStore/加密封装）；逐一对 `expect`/`actual` 看密钥来源、证书 Pinning、存储路径与权限；检查 Android/iOS 调用共享层时是否都做了入参校验。
- **防范**：共享层不信任任何“来自平台”的字符串/字节，在共享层或平台层做统一校验与白名单；加解密与存储的 expect/actual 双端均使用平台推荐安全 API（Keystore/Keychain、EncryptedSharedPreferences 等）；依赖的 KMP 库（Ktor、SQLDelight 等）及时跟进安全更新与配置（如禁用不安全的序列化、启用证书 Pinning）。

### 7. 二次漏洞与深层次问题

- **二次漏洞**：先通过一处漏洞写入或篡改数据（如评论、昵称、缓存），再在另一处逻辑中触发（如 SQL 注入、XSS、越权）。审计时需追踪“数据从哪来、到哪去”。
- **依赖与供应链**：第三方 SDK、native 库的已知 CVE；审计时结合 `dependency check`、CVE 库与版本升级建议。

### 8. 审计中的小技巧（移动端）

- **GPC 的等价**：移动端没有“魔术引号”，但要关注：服务端下发的数据是否被当成“可信”直接参与 SQL、命令、反射、反序列化；本地存储的数据是否在下次使用前再次校验。
- **编码与截断**：文件名、URL、Intent 参数中的编码与长度处理（如路径遍历、截断），尤其在文件读写、Intent 传递路径时。
- **多进程/多线程**：敏感状态是否在进程/线程间一致、是否有竞态（如支付、会话）。

## 第三部分 安全编程规范与体系建设

### 9. 输入与参数安全

- **原则**：所有来自外部的输入（用户输入、Intent、深链接、推送、剪贴板、文件、网络响应）均不可信，需校验与白名单。
- **实践**：类型与长度校验、业务规则校验（如金额>0）、避免将输入直接拼接到 SQL/命令/反射/反序列化；对 WebView 与 Scheme 做白名单。

### 10. 加密与密钥管理

- **存储**：密钥与长期凭证进 Keystore（Android）/ Keychain（iOS），不写死或放在普通文件/Preferences。
- **算法**：弃用 DES、RC4、MD5/SHA1（仅用于非安全场景）；对称用 AES-GCM，哈希用 SHA-256 及以上，密码存储用 Argon2/bcrypt 等。
- **传输**：HTTPS + 证书 Pinning，不在客户端实现自定义“加密协议”替代 TLS。

### 11. 功能安全设计要点（移动端）

| 功能 | 常见问题 | 安全设计要点 |
|------|----------|--------------|
| 登录/会话 | 弱密码、会话固定、Token 明文存储 | 强密码策略、服务端会话、Token 安全存储与刷新 |
| 找回密码/验证码 | 验证码可爆破、未绑定用户、凭证可预测 | 限次、限时、绑定身份、随机强凭证 |
| 支付 | 金额/数量客户端可改、重复提交 | 金额与订单以服务端为准、幂等与锁 |
| 文件/数据下载 | 路径遍历、任意文件读、未校验完整性 | 白名单路径、校验签名/哈希 |
| 深链接/推送 | 未校验来源与参数、跳转劫持 | Scheme/链接白名单、参数校验 |
| 生物识别 | 仅本地验证即通过、无服务端参与 | 与服务端二次校验、防重放 |
| **KMP 共享层** | 共享逻辑漏洞双端受影响、expect/actual 单端弱实现、平台入参未统一校验 | 优先审计 commonMain；expect/actual 双端等价安全；平台传入共享层的数据在共享层或两端统一校验 |

### 12. 应用安全体系建设（横向细化 + 纵深）

- **横向细化**：在每个入口与敏感点做校验与防护（输入校验、输出编码、权限最小化、日志脱敏），规则越细，单点被突破成本越高。
- **纵深防御**：假设前端/网络被突破，仍通过服务端校验、多因素、权限分级、审计日志等降低影响。
- **移动端落地**：  
  - 发布前：代码审计 + 依赖扫描 + 自动化安全测试（如 DAST/IAST 若适用）。  
  - 运行时：完整性校验、Root/越狱检测（仅作增强）、安全 SDK（如风控、反作弊）。  
  - 运维：版本与依赖管理、漏洞响应流程、灰度与回滚。

## 参考资源（移动安全）

- **OWASP Mobile Security Project**：MASVS、MSTG、Top 10 移动风险。
- **CWE / CVE**：按 CWE 查移动相关弱点，按依赖查 CVE。
- **各平台安全指南**：Android 安全最佳实践、Apple 安全文档与 App Transport Security。
- **自动化**：MobSF、Qark、Semgrep 规则库、各厂商移动安全产品。
- **KMP**：Kotlin 官方 [Kotlin Multiplatform](https://kotlinlang.org/docs/multiplatform.html) 文档、Ktor、kotlinx.serialization、SQLDelight 安全配置与最佳实践；共享层审计时结合上述关键字与 expect/actual 对照。

## 小结

作为移动开发 Leader，可将本文档作为**内部代码审计与安全设计**的对照表：

1. **准备**：统一构建与多版本环境，熟悉 Manifest/Info.plist 等“配置”，配备静态分析与抓包工具。  
2. **审计思路**：敏感 API 回溯、数据流正向追踪、功能点定向审计、关键路径通读。  
3. **漏洞类型**：存储、组件与 Intent、加密与网络、WebView、SQL/命令、逻辑与业务、二次漏洞与依赖。  
4. **规范与体系**：输入校验、安全加密与密钥管理、功能安全设计、横向细化与纵深防御。
5. **KMP 项目**：优先审计 commonMain；expect/actual 双端安全等价；共享层与平台层对入参统一校验；依赖的 KMP 库及时更新与安全配置。

将上述内容融入团队的 Code Review 清单、安全培训与发布流程，即可在移动端建立起与 Web 代码审计同等级别的安全能力。
