# 第 12 章 应用安全体系建设（横向细化 + 纵深）

> 本章对应主文档「第三部分 安全编程规范与体系建设」第 12 节，说明横向细化策略、纵深防御及在移动端的落地。

---

## 12.1 横向细化策略

### 12.1.1 含义

- **思想**：在**每一个**入口与敏感点都做校验与防护，不依赖「单点保险」；规则做得越细，单点被突破的成本越高，整体被突破的概率越低。
- **类比**：类似「木桶原理」的正面版——每一块板都尽量加高，而不是只加高一块。

### 12.1.2 在移动端的落地

| 层面 | 措施示例 |
|------|----------|
| **输入** | 所有外部输入（Intent、深链接、推送、剪贴板、文件、网络）在入口或共享层做类型、长度、白名单与业务校验。 |
| **输出** | 日志、界面、分享、第三方 SDK 不输出敏感信息；必要时脱敏与分级。 |
| **权限** | 仅申请业务必需的权限；运行时权限在用到时再申请并说明用途；定期审查权限使用情况。 |
| **存储** | 敏感数据进加密存储或 Keychain/Keystore；备份排除敏感内容。 |
| **网络** | 全站 HTTPS、严格证书校验、关键接口 Pinning；不信任用户 CA。 |
| **组件** | Android 组件最小化导出；Intent 参数白名单与校验；调用方签名/包名校验（如需）。 |
| **依赖** | 依赖清单、CVE 扫描、敏感 SDK 行为与权限审查；门禁与升级策略。 |

每个「入口」与「敏感操作」都应有对应的检查项与实现，而不是「只防一处」。

---

## 12.2 纵深防御策略

### 12.2.1 含义

- **思想**：假设**某一层已被突破**（如客户端被篡改、网络被窃听、单点校验被绕过），仍通过**其他层**的机制限制影响、发现异常或阻断攻击。
- **目的**：降低「单点失效即全盘失效」的风险；提高攻击者成本与可被发现概率。

### 12.2.2 在移动端的落地

| 层次 | 假设前一层被突破 | 本层措施示例 |
|------|------------------|--------------|
| **客户端** | - | 输入校验、安全存储、证书 Pinning、混淆与加固（增加成本）、Root/越狱检测（仅作增强）。 |
| **网络** | 客户端被篡改可发任意请求 | 服务端严格校验所有请求（身份、权限、参数、业务规则）；限速、限次、风控。 |
| **服务端** | 某接口被绕过或滥用 | 敏感操作多因素校验；权限分级与最小权限；审计日志与异常检测。 |
| **数据与运维** | 某账号或接口泄露 | 数据分级与脱敏；访问控制与审计；漏洞响应与补丁流程。 |

移动端自身可做的是「客户端层」的加固与合规；**纵深**主要体现在**服务端不信任客户端**、多因素、权限与审计等，需要前后端与运维一起建设。

---

## 12.3 移动端落地时间线

### 12.3.1 发布前

- **代码审计**：按本系列文档的审计思路与漏洞类型做白盒审计；对 KMP 项目优先审计 commonMain 与 expect/actual。
- **依赖扫描**：集成 OWASP Dependency-Check、Snyk 等，在 CI 中阻断已知高危 CVE。
- **自动化安全测试**：若有 DAST/IAST 或移动端专用扫描（如 MobSF 集成到 CI），在流水线中执行并设定门禁。
- **发布门禁**：未通过安全门禁（如高危漏洞未修复、依赖 CVE 未处置）的版本不允许发布生产。

### 12.3.2 运行时（可选增强）

- **完整性**：对关键 so/框架做完整性校验（可选）；注意与商店政策、用户体验的平衡。
- **Root/越狱检测**：可作为「增强」手段，限制部分敏感功能在已 root/越狱设备上的使用；不作为唯一安全依赖。
- **安全 SDK**：风控、反作弊、安全键盘等按业务需要集成；关注其隐私与权限合规。

### 12.3.3 运维与响应

- **版本与依赖管理**：维护当前线上版本的依赖清单与已知 CVE 状态；制定升级与回滚策略。
- **漏洞响应流程**：接收漏洞报告、评估影响、制定修复与沟通计划、发布补丁、复盘。
- **灰度与回滚**：发布与回滚能力；安全补丁可优先灰度或全量推送。

---

## 12.4 与团队流程的结合

- **Code Review**：将「输入校验、敏感存储、网络与加密、Intent/深链接」等纳入 Review 检查项；可引用本系列文档的章节作为 checklist。
- **安全培训**：对开发与测试进行移动安全与安全编码培训；新员工与外包接入前完成基础培训。
- **设计评审**：新功能或架构变更时进行安全设计评审（威胁建模、数据流、权限与接口设计）。
- **发布流程**：发布前必须通过安全门禁与审计/扫描结论确认；重大版本可安排专项安全测试。

---

## 12.5 小结

- **横向细化**：在每个入口与敏感点做校验与防护，不依赖单点；输入、输出、权限、存储、网络、组件、依赖等均有对应措施。
- **纵深防御**：假设某一层被突破，通过服务端校验、多因素、权限分级、审计与响应降低影响；移动端侧重客户端层，与后端共同形成纵深。
- **落地**：发布前（审计、依赖扫描、自动化测试、门禁）、运行时（可选增强）、运维（版本与依赖、漏洞响应、灰度与回滚）；与 Code Review、培训、设计评审、发布流程结合。
