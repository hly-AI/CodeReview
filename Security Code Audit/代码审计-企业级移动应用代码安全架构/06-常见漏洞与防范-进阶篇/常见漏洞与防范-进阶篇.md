# 第 6 章 常见漏洞与防范（进阶篇）

> 本章对应主文档「第二部分 漏洞挖掘思路与防范」第 6 节，详细展开 WebView 与前端协议、本地 SQL 注入与命令执行、逻辑与业务漏洞、KMP 架构相关风险的挖掘与防范。

---

## 6.1 WebView 与前端协议

### 6.1.1 漏洞现象

- **Android**：`addJavascriptInterface` 暴露的接口可被页面内 JS 调用，若接口未做校验或暴露敏感能力（如读文件、发请求、执行逻辑），则存在 XSS 或越权风险；`WebView.loadUrl`/`loadUrlWithData` 加载任意 URL 或 `file://`，导致钓鱼或本地文件泄露。
- **iOS**：`WKWebView` 的 `evaluateJavaScript` 若传入不可信字符串，可能执行恶意脚本；`loadRequest` 加载不可信 URL 或本地文件，存在类似风险。
- **深链接 / URL Scheme**：未校验 Scheme 与参数，可被第三方 App 调起并传入恶意参数，导致跳转劫持、参数注入或逻辑绕过。

### 6.1.2 挖掘方法

- **搜索**：WebView 相关 API、`@JavascriptInterface`、`shouldOverrideUrlLoading`、`onLoadResource`、`evaluateJavaScript`、`loadRequest`、URL Scheme 注册与处理（Android Intent 的 `data`/`scheme`，iOS 的 `openURL`/`application(_:open:options:)`）。
- **关注**：JS 接口暴露的方法是否做入参校验；加载的 URL 是否白名单、是否允许 `file://`；Scheme 参数是否直接拼接到 URL 或用于内部逻辑。

### 6.1.3 防范要点

- **JS 接口**：最小化暴露接口；对入参做类型与白名单校验；不将用户可控字符串直接传入 `evaluateJavaScript`。
- **URL 与 Scheme**：仅加载白名单域名或路径；禁用不必要的 `file://` 访问；Scheme 与参数校验后再用于跳转或业务逻辑。
- **深链接**：校验来源（如仅处理自家域名或签名）；参数做白名单与业务校验。

## 6.2 本地 SQL 注入与命令执行

### 6.2.1 SQL 注入（本地）

- **现象**：SQL 语句拼接用户输入（包括 Intent、文件内容、网络响应、本地存储），未使用参数化查询或 ORM 安全接口，导致 SQL 注入（本地库可被读/写/删）。
- **挖掘**：搜 `rawQuery`, `execSQL`, `query` 与字符串拼接、`String.format` 拼 SQL、以及 SQL 语句中的 `+`/`concat` 与用户输入。
- **防范**：使用参数化查询（`?` 占位符、`SQLiteStatement`）或 ORM；不将不可信字符串拼入 SQL。

### 6.2.2 命令执行

- **现象**：`Runtime.exec`、`ProcessBuilder`（Android）或 `Process`/`NSTask`（iOS）的参数来自用户输入、Intent 或文件，导致命令注入或任意命令执行。
- **挖掘**：搜 `Runtime.getRuntime().exec`, `ProcessBuilder`, `Process`，追踪参数来源。
- **防范**：禁止用户输入直接进入命令行；若必须调用外部命令，使用白名单命令与参数，并对参数做转义（如 `escapeshellarg` 等价逻辑）。

## 6.3 逻辑与业务漏洞

### 6.3.1 漏洞现象

- **支付**：金额、数量、订单状态由客户端传入且服务端未严格校验，可被篡改（如改金额为 0.01、改数量为负数或极大值、重复提交同一订单）。
- **找回密码/验证码**：验证码可爆破（无错误次数/时间限制）、验证凭证未绑定用户或可预测、验证码在响应或前端明文返回。
- **投票/抽奖/活动**：参与次数、资格依赖客户端或 Cookie 可伪造，删除 Cookie 或改包即可重复参与。

### 6.3.2 挖掘方法

- **通读**支付、找回密码、验证码、投票/活动相关代码，看「谁说了算」：金额、数量、次数、资格是否由**服务端**计算并校验；客户端是否仅做展示与提交，服务端是否做幂等与并发控制。
- **抓包**：修改请求参数（金额、数量、状态、用户 ID）重放，观察服务端是否拒绝或返回错误。

### 6.3.3 防范要点

- **关键结果以服务端为准**：金额、库存、订单状态、参与次数等由服务端计算并持久化，客户端仅展示与提交；服务端校验与业务规则一致。
- **敏感操作多因素**：支付、改密码、改手机等结合验证码/二次验证；验证码限次、限时、绑定用户。
- **限速与绑定身份**：接口限速、参与次数与账号/设备绑定，不依赖客户端或 Cookie 单方面声明。

## 6.4 KMP 架构相关风险

### 6.4.1 漏洞现象

- **共享层（commonMain）**：加解密、网络、序列化、存储等存在弱实现或硬编码；某端 `actual` 实现与另一端安全等级不一致（如一端用 Keystore、另一端明文）。
- **平台入参未统一校验**：Android 或 iOS 传入共享层的 URL、路径、用户输入等未在共享层或两端统一校验，导致双端或单端可利用（如注入、越权）。

### 6.4.2 挖掘方法

- **commonMain**：重点审所有接触「外部输入」或「平台传入」的模块（Ktor HttpClient、kotlinx.serialization、SQLDelight、自定义 KeyValueStore/加密封装）；搜索 `expect`/`actual`，逐项核对密钥来源、证书 Pinning、存储路径与权限。
- **调用链**：Android Activity / iOS ViewController 调用共享层时，是否都做了入参校验；是否存在「仅一端校验」的情况。

### 6.4.3 防范要点

- **共享层不信任平台入参**：在共享层或平台层做统一校验与白名单；不将「来自平台」的字符串/字节直接用于 SQL、命令、反序列化、文件路径、URL 加载。
- **expect/actual 双端等价安全**：加解密与存储的 actual 均使用平台推荐安全 API（Keystore/Keychain、EncryptedSharedPreferences）；不在某一端弱化实现。
- **依赖与配置**：KMP 库（Ktor、SQLDelight 等）及时跟进安全更新；禁用不安全的序列化选项、启用证书 Pinning 等安全配置。

---

## 6.5 小结

- **WebView 与 Scheme**：最小化 JS 接口、URL 与 Scheme 白名单、禁止不必要的 file 访问。
- **SQL 与命令**：参数化/ORM、禁止用户输入进命令行；白名单与转义。
- **逻辑与业务**：服务端为准、多因素、限速与身份绑定。
- **KMP**：共享层与平台入参统一校验；expect/actual 双端安全等价；依赖与安全配置及时更新。
