# 第 4 章 通用代码审计思路（移动端）

> 本章对应主文档「第二部分 漏洞挖掘思路与防范」第 4 节，详细展开四种审计思路及 KMP 下的侧重，并给出敏感关键字列表。

---

## 4.1 四种审计思路概述

与 Web 代码审计类似，移动端可归纳为四种互补思路：

1. **敏感 API / 关键字回溯**：从「危险调用」反推输入来源，判断是否可控、是否校验。
2. **可控变量正向追踪**：从入口（Intent、用户输入、存储、网络响应等）跟踪到系统 API 或业务敏感逻辑。
3. **敏感功能点定向审计**：按功能模块（登录、支付、找回密码等）逐块读代码。
4. **通读关键路径**：从启动 → 登录 → 核心业务流程，理解架构与数据流，再结合上述方式挖洞。

---

## 4.2 敏感 API / 关键字回溯

### 4.2.1 思路说明

- **做法**：在工程中搜索「危险 API」或关键字（如加解密、文件读写、Intent、WebView、网络、SQL、反射、JNI），定位到调用点后，**反向追踪** 传入参数的数据来源。
- **判断**：若来源为「用户输入、Intent、剪贴板、网络响应、本地存储且可被篡改」等可控数据，且未经严格校验或白名单，则存在潜在漏洞。
- **优点**：可定向、高效地发现「API 使用不当」类问题；适合作为首轮扫描手段。
- **缺点**：对「纯逻辑漏洞」「多步组合」「设计缺陷」覆盖不足，需结合其他思路。

### 4.2.2 典型危险 API 分类

| 类别 | Android 示例 | iOS 示例 |
|------|--------------|----------|
| 存储 | `getSharedPreferences`, `openFileOutput`, `FileOutputStream` | `UserDefaults`, `FileManager`, `write(to:)` |
| 加密/哈希 | `Cipher.getInstance`, `MessageDigest`, `SecretKeySpec` | `CCCrypt`, `SecKeyCreate`, `CryptoKit` |
| 网络 | `HttpURLConnection`, `OkHttpClient`, `Retrofit` | `URLSession`, `NSURLConnection` |
| 组件/Intent | `startActivity`, `sendBroadcast`, `getIntent()`, `getExtras()` | `openURL`, `application(_:open:options:)` |
| WebView | `loadUrl`, `evaluateJavascript`, `addJavascriptInterface` | `loadRequest`, `evaluateJavaScript` |
| 数据库 | `rawQuery`, `execSQL`, `SQLiteDatabase` | `sqlite3_*`, `execute` |
| 命令/进程 | `Runtime.exec`, `ProcessBuilder` | `Process`, `NSTask` |
| 反射/动态 | `Class.forName`, `Method.invoke`, JNI | `NSClassFromString`, `performSelector` |

---

## 4.3 可控变量正向追踪

### 4.3.1 思路说明

- **做法**：先找出「所有接收外部输入」的入口（Activity/Fragment 入参、Intent、深链接、推送、剪贴板、文件、网络回调），再**正向**跟踪这些数据在代码中的传递与使用。
- **关注**：是否被传入敏感 API（如 SQL、命令、反射、反序列化、文件路径、WebView URL）；在传递过程中是否被校验、转义或白名单过滤。
- **优点**：能覆盖「从入口到 sink」的完整数据流，不易遗漏某一入口；对「多入口、一 sink」的场景尤其有效。
- **缺点**：入口多时工作量大，可先做「关键入口 + 关键 sink」的抽样或工具辅助（如数据流分析）。

### 4.3.2 常见入口

- **Android**：`Activity.onCreate` 中 `intent?.extras`、`Activity.getIntent()`、`SharedPreferences`、`ContentProvider` 的 query/insert、从文件或 SD 卡读取的路径与内容。
- **iOS**：`UIApplicationDelegate` 的 `open URL`、`UserDefaults`、剪贴板、从文件或网络加载的数据。
- **通用**：网络响应（JSON/XML 等）解析后的字段、推送 payload、来自第三方 SDK 的回调参数。

---

## 4.4 敏感功能点定向审计

### 4.4.1 思路说明

- **做法**：根据经验列出「易出问题」的功能（登录、支付、找回密码、分享、文件上传/下载、深链接、推送、本地存储、生物识别等），按功能模块找到对应代码，**整块阅读** 该功能的实现。
- **关注**：业务逻辑是否正确（如金额/数量是否由服务端决定；验证码是否限次、限时、绑定用户）；是否有越权、绕过、可预测凭证等问题。
- **优点**：针对性强，适合「时间有限、优先保障核心业务」的审计。
- **缺点**：依赖审计者对「常见漏洞出现在哪些功能」的经验；可能遗漏非典型功能或底层通用模块。

### 4.4.2 功能与常见风险映射（简要）

| 功能 | 常见风险 |
|------|----------|
| 登录/会话 | 弱密码、会话固定、Token 明文存储、撞库 |
| 找回密码/验证码 | 验证码可爆破、未绑定用户、凭证可预测、短信轰炸 |
| 支付 | 金额/数量客户端可改、重复提交、未校验订单状态 |
| 文件/下载 | 路径遍历、任意文件读、未校验完整性 |
| 深链接/推送 | 未校验来源与参数、跳转劫持、任意 Scheme |
| 生物识别 | 仅本地验证即通过、无服务端参与、重放 |

## 4.5 通读关键路径

### 4.5.1 思路说明

- **做法**：从 App 启动入口（如 MainActivity、AppDelegate）开始，沿「启动 → 闪屏/引导 → 登录 → 核心业务流程」读下去，理解**整体架构、模块划分、数据流与信任边界**。
- **目的**：建立全局观，便于后续用「回溯」或「正向追踪」时快速定位到正确模块；同时发现「设计层面」的信任假设错误（如某层默认信任下层数据）。
- **适用**：企业自研 App 的全量审计、新项目接手、需要输出「架构与数据流图」时。

### 4.5.2 阅读顺序建议

1. 入口与路由：谁在何时启动哪个界面/模块。
2. 登录与鉴权：Token 如何获取、存储、携带；登出与过期处理。
3. 核心业务：关键业务流程涉及哪些网络请求、本地存储、第三方调用；敏感操作是否有服务端二次校验。
4. 共享/基础设施：如 KMP 的 commonMain、公共网络层、存储封装、加解密封装，供哪些模块使用。

## 4.6 KMP 架构下的审计侧重

### 4.6.1 先审共享层（commonMain）

- 加解密、网络请求、序列化/反序列化、本地存储封装、Token 处理等若在 **commonMain** 实现，一旦有问题会**同时影响 Android 与 iOS**，应优先做敏感 API 回溯与数据流追踪。
- 重点：所有接收「来自平台」的字符串/字节的接口（如平台传入的 URL、路径、用户输入），是否在共享层或平台层做了校验与白名单。

### 4.6.2 expect/actual 对照审

- 对每个 `expect` 的 **平台 actual** 实现做核对：密钥来源、证书 Pinning、存储路径与权限、是否使用 Keystore/Keychain 等安全 API。
- 避免「一端实现正确、另一端弱实现或硬编码」导致单端漏洞。

### 4.6.3 入口在各自平台

- Intent、深链接、剪贴板、文件访问等仍由 **Android/iOS 端** 接收入参，再传入 KMP 层。
- 审计时追踪「平台入口 → 共享层」：校验是在共享层统一做，还是在各端分别做；若分别做，是否两端都做了，避免遗漏一端。

## 4.7 敏感关键字示例（便于搜索）

### 4.7.2 iOS
