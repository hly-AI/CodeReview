# 第 1 章 为什么需要移动端代码审计

> 本章对应主文档「第一部分 代码审计前的准备」第 1 节，说明移动端代码审计的必要性、适用场景与能力要求。

---

## 1.1 企业安全与合规诉求

### 1.1.1 监管与等保要求

- **等级保护**：金融、政务、医疗、教育等行业 App 常需满足等保 2.0 中对应用安全的要求，其中**安全设计与代码实现**的审计是测评项之一。
- **行业规范**：如支付类 App 需符合 PCI DSS、各监管机构对客户端安全与数据保护的指引，代码审计是证明「安全措施落地」的重要依据。
- **应用市场上架**：苹果 App Store、Google Play 及国内应用市场对隐私、权限、安全漏洞有明确要求；历史漏洞与恶意行为可导致下架或拒绝上架，代码审计有助于在上架前发现并修复问题。

### 1.1.2 内部安全运营

- **SDL（安全开发生命周期）**：在需求、设计、编码、测试、发布、运维各阶段嵌入安全活动；**白盒测试（代码审计）** 主要落在「测试」阶段，与设计阶段的安全评审、发布前的渗透测试形成互补。
- **供应链与外包**：引入第三方 SDK、外包开发、并购带来的代码资产，需要通过代码审计评估风险并制定整改或隔离策略。
- **事件响应**：一旦出现疑似泄露、篡改或入侵，对相关模块做定向代码审计可快速定位根因、评估影响范围并指导修复与复盘。

---

## 1.2 SDL 在移动端的落地

### 1.2.1 安全贯穿全生命周期

| 阶段     | 安全活动示例 |
|----------|--------------|
| 需求分析 | 隐私与合规需求、敏感数据分类、威胁建模输入 |
| 设计     | 安全架构评审、接口与数据流设计、认证与授权模型 |
| 编码     | 安全编码规范、依赖管理、敏感信息不落库/不写死 |
| **测试** | **代码审计（白盒）、依赖扫描、渗透测试（黑盒）、自动化安全测试** |
| 发布     | 发布门禁（无高危漏洞）、版本与依赖记录 |
| 运维     | 漏洞响应、补丁与升级策略、日志与监控 |

代码审计是「测试」阶段中**从源码层面发现缺陷**的核心手段，与黑盒渗透、自动化扫描互为补充：代码审计能发现逻辑漏洞、设计缺陷和「仅看行为难以触发」的问题。

### 1.2.2 代码审计在移动端的价值

- **可控与可复现**：直接针对源码，不依赖运行环境是否可触达；问题可精确定位到文件/行，便于修复与回归。
- **覆盖共享逻辑**：在 KMP、RN、Flutter 等方案下，共享层一处问题可能影响多端，代码审计能系统梳理数据流与信任边界。
- **合规与举证**：审计报告与修复记录可作为「已实施安全措施」的证明，支撑等保、合规审查与客户/合作方沟通。

---

## 1.3 渗透测试与应急响应场景

### 1.3.1 渗透测试中的代码审计

- **信息增强**：在授权渗透中，若获得脱壳/反编译包或部分源码，可通过代码审计快速定位敏感接口、硬编码密钥、调试接口、隐藏功能，指导后续攻击面测试。
- **Payload 与绕过**：针对证书 Pinning、Root/越狱检测、签名校验等，结合代码逻辑设计绕过或构造更精准的测试用例。
- **漏洞验证**：扫描器或黑盒发现的疑似点（如 SQL 注入、越权），通过阅读相关代码确认是否真实可利用及影响范围。

### 1.3.2 应急响应

- **疑似泄露或篡改**：审计与「数据流、存储、网络」相关的代码，确认敏感数据是否加密、是否被错误输出到日志/界面/第三方。
- **已知 CVE 或供应链事件**：根据依赖清单定位到具体模块与调用链，评估是否受影响、是否需要升级或临时缓解。
- **恶意行为分析**：对疑似被篡改的 App 或 SDK 做代码级分析，识别后门、异常网络请求、权限滥用等。

---

## 1.4 移动端代码审计的能力要求

### 1.4.1 编程与语言基础

- **至少能读懂代码**：理解业务逻辑、数据流、控制流；能区分「用户/外部可控输入」与「内部/可信数据」。
- **语言与平台**：
  - **Android**：Java/Kotlin，了解 Activity、Intent、SharedPreferences、网络与存储常用 API。
  - **iOS**：Swift/Objective-C，了解 ViewController、UserDefaults、Keychain、URLSession 等。
  - **KMP**：Kotlin 共享代码、expect/actual、常见 KMP 库（Ktor、serialization、SQLDelight 等）。
  - **其他跨平台**：若涉及 RN、Flutter，需对应掌握 JS/TS 或 Dart 及与原生桥接方式。

### 1.4.2 安全与漏洞知识

- **常见漏洞类型**：不安全存储、组件暴露、弱加密与硬编码、中间人、WebView 与深链接、SQL/命令注入、逻辑与业务漏洞等（详见本系列第 5、6 章）。
- **数据流与信任边界**：能追踪「输入从哪来、经过哪些处理、最终用到哪」，判断是否未经校验即参与敏感操作。
- **平台差异**：Android 与 iOS 在权限、组件模型、存储与加密 API 上的差异，以及 KMP 共享层与各端实现的对应关系。

### 1.4.3 工具与习惯

- **检索与阅读**：熟练使用 IDE 全局搜索、调用层级、符号跳转；能按「敏感 API、关键字、功能入口」快速定位代码。
- **辅助工具**：静态分析（MobSF、Semgrep、SonarQube 等）、抓包（Charles/Burp）、依赖与 CVE 扫描（见第 3 章），并能结合人工判断排除误报、确认真实漏洞。

---

## 1.5 小结

- 移动端代码审计是**企业合规、SDL 测试阶段、渗透与应急**中的重要手段。
- 与黑盒测试、自动化扫描互补，能系统发现逻辑漏洞、设计缺陷与多端共享层风险。
- 需要具备**读代码、懂漏洞、熟悉平台与工具**的能力；KMP 等跨平台架构下，需额外关注共享层与 expect/actual 的审计。
